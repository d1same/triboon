## Decisions of record (made on delegated judgment, 2026-06-11)

1. **Speed model**: prefetch-on-title-open (Phase 2 wiring); Play = mount + burst. ~1s perceived.
2. **Native, zero-dep, fully seekable**: store-mode RAR4 + RAR5 (the scene standard for video,
   ~95%+ of releases) and store-mode ZIP. Multi-volume with both naming schemes.
3. **Detection + verdict, not playback, for the rest**: compressed RAR (method bits), encrypted
   (data- and header-encrypted), 7z, are *detected* and surfaced as verdicts/tags (🐢 or
   blocked) so ranking skips or deprioritizes them. Rationale: pure-JS RAR decompression is a
   multi-week codec port; sidecar extractors (unrar/7z) cannot seek a remote virtual file, so
   they'd force spool-to-disk ("download then play"), which betrays the speed value. Raise
   compressed-playback options with the owner at the Phase 1 demo.
4. **Passwords**: parse `<meta type="password">` from NZB head; detect encryption flags; defer
   decryption (RAR5 PBKDF2+AES is nativly feasible later; needs the compressed story first).
5. **Multi-provider failover**: NntpPool accepts a provider list; per-article failover on 430.
6. **Fixtures**: hand-rolled clean-room store-RAR4/RAR5/ZIP writers in test helpers (validated
   against real unrar in a throwaway Docker container); compressed/password/7z fixtures
   generated by real rar/7z in Docker and committed as binaries (no WinRAR/7-Zip on the dev box).

## Goal

Stream video from inside RAR (and 7z) archives in a mounted NZB, with byte-exact HTTP Range
semantics and working seek, judged against time-to-first-frame. Done = owner streams a real
RAR'd NZB from Easynews in VLC with working seeking, and says so.

## Speed model (clarified 2026-06-10)

Current note (2026-06-21): this Phase 1 speed model remains the reason for
prefetch and bounded health checks, but live runtime tuning moved to the
capacity model in `docs-streaming-performance.md`. Do not treat the old Easynews
16-connection benchmark as the current read-ahead rule.

Owner wants a perceived **press-Play → first frame within ~1 second**. The Stremio +
UsenetStreamer feel comes from resolving sources *before* play (while the user looks at the
title page). Triboon adopts the same trick: indexer search + triage prefetch when a title is
opened/focused; pressing Play only does mount (ms) + first-segment burst (~150–250ms measured,
according to the historical local-only benchmark notes). Cold worst case (no
prefetch possible) stays bounded < 5s.

## Devil's-advocate findings (assumptions to verify, ordered by risk)

1. **Compressed RAR seeking** — store (m0) dominates video releases, but compressed (m1–m5)
   and solid archives exist; seeking there means decompressing from the block/solid-run start.
   Plan: 🐢-tagged, decode-ahead cursor, never block the picker on it.
2. **Multi-part volume boundaries** — a single read can span .rar → .r00; each volume has its
   own local headers so archive-offset → volume-offset mapping must be computed per volume, and
   both old (.r00) and new (.part01.rar) naming must work. NZB declared sizes can lie — offsets
   must come from actual yEnc =ybegin/=ypart headers, not NZB metadata.
3. **RAR4 vs RAR5** — different header formats; both are in the wild. Both must parse.
4. **Password-protected** — two flavors: encrypted file data (listable) vs encrypted headers
   (not even listable). Some NZBs carry the password in `<meta type="password">`. Open question.
5. **7z is the odd one out** — its directory lives at the *end* of the file (tail fetch needed
   before anything streams) and solid LZMA blocks make seeking awful. Rare for video. Open
   question on scope.
6. **Obfuscated releases** — inner file names may be junk; primary-pick must use size +
   signature sniffing, not extension alone.
7. **Easynews STAT-miss penalty (~555ms)** — health gate redesign: parallel STATs confirm
   healthy in ~60–250ms; 500ms soft timeout marks suspect → auto-advance. Never wait for misses.
